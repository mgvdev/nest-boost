# Skill Builder

nest-boost cannot ship a skill for every library. This skill teaches you to build one
on demand and persist it in the project's knowledge base so every configured agent —
and every future session — benefits.

## When to use this skill
- The user gives you a dependency and asks for a skill / guidance for it.
- You find yourself repeatedly looking up the same library's API.

## The knowledge base
Custom skills live in **`.nest-boost/skills/<skill-name>/SKILL.md`**. This directory is
the source of truth and should be **committed** to git. Running `nest-boost update` copies
these into each configured agent's skill directory (`.claude/skills/`, Cursor, …). A local
skill with the same name as a packaged one overrides it.

## Procedure

### 1. Identify the package + version
Use the `nest-boost` `application_info` MCP tool (or read `package.json`) to get the exact
package name and installed version. Skills must match the version actually installed.

### 2. Gather authoritative sources (in priority order)
1. **The library's own LLM docs** — try `https://<docs-domain>/llms.txt` (and `llms-full.txt`).
   Many modern libs publish one (e.g. Better Auth). Fetch it, find the relevant page, read it.
2. **Official docs + README** — the docs site, and the npm/GitHub README for setup snippets.
3. **Context7 MCP** (if available) — `resolve-library-id` then `query-docs` for current API.
4. **An existing community skill** — try `npx skills add <lib>/skills`; if it exists, prefer
   pulling it and skip authoring. (Check https://agentskills.io.)
5. **NestJS integration guide** — if the lib has one, it takes precedence for how to wire it
   into modules/DI. Use the `nestjs-docs` skill for framework-side details.

Prefer fetched docs over memory. Note the version — APIs drift.

### 3. Focus the content on *NestJS* usage
A good library skill answers: how to register it in a module (`forRoot`/`forRootAsync`/
`forFeature`), what to inject and via which token/decorator, required config, common
patterns, gotchas, and how to test it. Keep it concise and actionable — patterns, not prose.

### 4. Write the skill
Create `.nest-boost/skills/<skill-name>/SKILL.md`. Name it `<lib>-development` or
`<lib>-integration` (kebab-case, no scope, e.g. `drizzle-development`,
`nestjs-schedule-integration`). Use this shape:

```markdown
---
name: <skill-name>
description: <one line — what it helps build, and when to use it>
---

# <Library> in NestJS

## When to use this skill
Use when <task with this library>.

## Setup
- Install: `<packages>`
- Register: `<Module.forRoot(...)>` in the owning module.

## Patterns
- <inject X via @Inject(TOKEN) / @InjectY()>
- <core usage snippet>

## Rules
- <do / don't specific to this lib in Nest>

## Ground yourself
Use the `nest-boost` `module_graph` / `list_routes` tools to see how it's wired.
```

You may add supporting files (references, templates) in the same folder if useful.

### 5. Add it to the knowledge base
Propagate to every configured agent:

```bash
bunx nest-boost update
```

Then tell the user to commit `.nest-boost/skills/` (source of truth). The copies under
`.claude/skills/` etc. are regenerated by `nest-boost update` and can be gitignored.

### 6. Verify
Confirm `.nest-boost/skills/<name>/SKILL.md` exists and that `nest-boost update` reported it
under the configured agents. Sanity-check the skill by using it for the task that prompted it.

## Rules
- One skill per library concern; keep each focused.
- Don't invent API — cite what you fetched; when unsure, fetch again.
- Always target the installed version.
- Never write secrets or project-specific credentials into a skill.
