# Configuration — `nest-boost.json`

Written by `install` at the project root and read by `update` and the MCP server.

```json
{
  "projects": [
    { "name": "api", "type": "application", "root": "apps/api", "entryModule": "apps/api/src/api.module.ts", "moduleExport": "ApiModule" },
    { "name": "shared", "type": "library", "root": "libs/shared" }
  ],
  "defaultProject": "api",
  "agents": ["claude"],
  "architecture": "hexagonal",
  "auth": "better-auth",
  "testLayout": "central",
  "database": { "url": "postgres://user:pass@localhost:5432/app" },
  "evaluate": { "enabled": true }
}
```

## Fields

| Field | Type | Notes |
| --- | --- | --- |
| `projects` | array | One entry per workspace project. A single-app project has one `application` with `root: "."`. |
| `projects[].name` | string | Project name (from `nest-cli.json`, or `"app"` for a single app). |
| `projects[].type` | `"application"` \| `"library"` | Libraries are listed but never booted. |
| `projects[].root` | string | Repo-relative project root. |
| `projects[].entryModule` | string | Path to the file exporting the root module (applications). |
| `projects[].moduleExport` | string | Named export of the root module. Default `AppModule`. |
| `defaultProject` | string | Project the MCP boots when no `project` argument is given. |
| `agents` | string[] | Agents configured during install; `update` re-targets these. |
| `architecture` | string | `standard` \| `cqrs` \| `hexagonal`. Drives a guideline + skill. |
| `auth` | string | `none` \| `passport` \| `better-auth`. Drives a guideline + skill. |
| `testLayout` | string | `colocated` \| `colocated-subfolder` \| `central`. Drives a guideline. |
| `database` | object | Optional. `{ "url": "..." }` for the database tools. |
| `evaluate` | object | Optional. `{ "enabled": false }` to disable the `evaluate` tool. Enabled by default. |

## Backward compatibility

An older config with a top-level `entryModule` / `moduleExport` (pre-workspace schema) is
migrated on read into a single `projects` entry, so upgrading is seamless. Re-running
`install` writes the current schema.

## What to commit

Everything nest-boost generates (`nest-boost.json`, `.mcp.json`, `CLAUDE.md`, `.claude/skills`,
…) can be gitignored — it's regenerated by `install`/`update`. The exception is
`.nest-boost/skills/`, your custom-skill knowledge base, which **should** be committed. See
[Guidelines & skills](./guidelines-and-skills.md).
